   - Orders 흐름과 무관하게 판매자가 임의로 SELLING/RESERVED/SOLD를
     직접 바꿔버릴 수 있어서, 단계 1의 상태전이 로직을 도입해도 이 경로가
     막히지 않으면 정합성이 깨짐

2. Orders에 상태전이 로직이 없음 (이번 재구성의 핵심 이슈)

     PAY_COMPLETED / COMPLETED 5개 값이 이미 정의되어 있음
   - 하지만 OrdersService.save()는 항상 PAY_COMPLETED로 즉시 생성하고
     Item.status를 RESERVED로 바꿔버림 -> REQUESTED/ACCEPTED/CANCELED는
     정의만 되고 실제로는 아무 경로도 도달하지 못하는 죽은 값
   - 판매자 승인/거절, 구매자 취소 같은 흐름을 표현할 API가 없음

3. Orders에 협상 금액 개념이 없음
   - agreedPrice 같은 필드가 없어 항상 item.price를 그대로 사용
   - 가격 협상 결과를 남길 곳이 없음

4. 위시리스트 백엔드가 완전히 없음
   - frontend/src/pages/Wishlist.jsx, frontend/src/api/client.js의
     listWishlist() / toggleWishlist(itemId)는 이미 존재하지만
     대응하는 백엔드 엔티티/컨트롤러가 전혀 없음 (mock 모드에서만 동작)

5. 채팅/WebSocket 관련 코드, 엔티티, 의존성이 전혀 없음
   - build.gradle에 WebSocket/STOMP 관련 의존성 없음
   - domain/chat 패키지, ChatRoom/ChatMessage 엔티티 등 어떤 관련 코드도
     없음 (2026-07-16 시점과 동일, 변경 없음)

6. LoginCheckInterceptor의 인증 우회 로직 - 완화되었으나 완전 해소는 아님
   - method.equals("GET") && requestURI.contains("items") 였던 조건이
     requestURI.startsWith("/api/items")로 개선됨
   - 다만 여전히 AntPathMatcher 기반 정교한 경로 패턴 매칭이 아니라
     문자열 startsWith이므로, 예를 들어 향후 /api/items-report 같은
     경로가 생기면 의도치 않게 로그인 없이 공개될 여지가 남아 있음
   - 심각도는 최초 버전 대비 하향 조정 (완전 해소는 아니므로 계속 인지)

7. GlobalExceptionHandler가 아직 얕음
   - 여전히 도메인 예외 1개당 @ExceptionHandler 1개, 상태코드 고정 매핑
     방식 그대로 (DuplicateMemberException/MemberException/ItemException/
     LoginFailException/UnauthorizedException 5개뿐)
   - Orders 도메인은 전용 예외 클래스 없이 기존 ItemException/
     MemberException을 재사용 중 ("구매할 수 없는 상품입니다" 등도
     ItemException으로 던짐) -> 신규 도메인이 늘어날수록 이 예외 설계
     부채가 계속 전파되고 있음
   - "로그인은 했지만 권한 없음"(403) 개념 여전히 없음 (401만 존재)
   - "상태전이 위반"(409) 케이스도 여전히 없음 (애초에 상태전이 로직
     자체가 없어서 발생할 일도 없었음 -> 단계 1 구현 시 반드시 필요해짐)

8. Item.status와 Orders.orderStatus를 억지로 하나로 합치려 하면 안 됨
   - Item.status = "상품 카탈로그 뷰에 보여지는 상태"
   - Orders.orderStatus = "개별 주문 건의 생애주기"
   - 역할을 분리해서 설계할 것 (최초 버전과 동일한 원칙, 계속 유효)

확정된 설계 방향
================================================================

- Transaction을 새로 만들지 않고 기존 Orders 엔티티를 확장한다
  (buyer, item, orderStatus는 이미 있음 -> agreedPrice 필드를 추가하고,
   이미 정의된 OrderStatus 값들을 실제로 순회하는 상태전이 로직을 도입)

- seller는 별도 필드로 비정규화하지 않고 기존처럼 item.member를 통해
  간접 참조한다 (Orders에 seller 필드를 새로 추가할 만큼 조회 패턴이
  복잡하지 않음 -> 불필요한 중복 필드 추가 지양)

- 구현 순서: Orders 상태전이 로직 -> 위시리스트 백엔드 -> 채팅
  (검색은 이미 완료되어 순서에서 제외. 채팅은 WebSocket 인증 연동 등
   인프라 리스크가 가장 크므로 여전히 마지막 순서)

- 채팅은 최초 버전과 동일하게 MVP 없이 처음부터 WebSocket/STOMP
  실시간 채팅으로 구현한다


================================================================
단계 1 - Orders 상태전이 로직
================================================================

목표: OrderStatus에 이미 정의돼 있지만 미사용인 REQUESTED/ACCEPTED/
CANCELED를 실제로 순회하는 상태전이 API를 추가하고, 협상 금액을
남길 수 있게 한다.

[Orders 엔티티 변경]

- agreedPrice (Integer) 필드 추가 - 상품 정가와 별도로 합의된 금액
  (없으면 item.price를 그대로 사용)

[API] (기존 OrdersController 패턴 참고)

기존 POST /api/orders/{itemId} (즉시구매, PAY_COMPLETED로 바로 생성)는
프론트(Checkout.jsx)가 이미 사용 중이므로 하위 호환을 유지한다.
이 위에 아래 상태전이 API를 추가로 얹는다.

PATCH /api/orders/{orderId}
  - 상태 전이 (요청자가 buyer/seller 중 해당 액션 권한을 가진 쪽인지 검증)
    판매자 승인    REQUESTED  -> ACCEPTED
    구매확정       ACCEPTED   -> PAY_COMPLETED / COMPLETED
    취소           (buyer/seller 누구든) -> CANCELED, 취소 시 Item.status는
                    SELLING로 복귀
  - 정해진 경로(REQUESTED -> ACCEPTED -> PAY_COMPLETED/COMPLETED,
    언제든 -> CANCELED)를 벗어난 역행/스킵은 OrdersException + 409

※ 즉시구매(POST) 흐름을 그대로 둘지, REQUESTED부터 시작하도록 바꿀지는
  구현 시점에 프론트 영향(Checkout.jsx가 즉시 완료를 기대하고 있음)을
  다시 확인하고 결정한다 - 이 로드맵에서는 결정 보류 항목으로 남긴다.

[검증 규칙]

- 상태 전이는 같은 @Transactional 경계 안에서 Item.status도 함께 갱신
  (기존 원칙 유지, 별도 도메인 이벤트 불필요)
- OrdersException 신규 도입, GlobalExceptionHandler에 409 매핑 추가
  (상태전이 위반 케이스 전용)

단계 4 - 채팅 (WebSocket/STOMP)
================================================================

목표: ChatRoom/ChatMessage 스키마를 설계하고 실시간 전송 계층을 얹는다.
(최초 버전과 동일, 변경 없음)

[신규 엔티티]

ChatRoom (domain/chat 패키지)
  - id
  - item   (@ManyToOne)
  - seller (@ManyToOne Member)
  - buyer  (@ManyToOne Member)
  - createdDate
  - (item_id, buyer_id) 조합에 유니크 제약을 걸어서 같은 상품에 대해 같은
    구매자가 채팅방을 중복 생성하지 않도록 함
  - ChatRoom은 Orders와 독립적으로 생성 가능 (구매자가 "채팅하기"만
    눌러도 방은 생김). 주문이 시작되면 선택적으로 어느 채팅방에서
    비롯됐는지 참조만 걸어둘 수 있음 (필수 아님)

ChatMessage
  - id
  - chatRoom (@ManyToOne)
  - sender   (@ManyToOne Member)
  - content
  - sentAt

[신규 의존성 및 설정]

- build.gradle에 spring-boot-starter-websocket 추가
- WebSocketConfig (@Configuration + @EnableWebSocketMessageBroker):
  STOMP 엔드포인트 등록, 메시지 브로커 설정
- *** 이 단계의 핵심 리스크: 세션 인증 연동 ***
  기존 LoginCheckInterceptor(HandlerInterceptor)는 WebSocket 핸드셰이크에
  재사용 불가. 별도의 HandshakeInterceptor를 만들어 HTTP 세션의
  LoginMember를 WebSocket 세션 attribute로 복사하고, 없으면 핸드셰이크를
  거부해야 함

[API]

POST /api/items/{itemId}/chatrooms
  - 채팅방 생성/조회 (이미 있으면 기존 방 반환), 로그인 필요

STOMP:
  /pub/chat/message        (발행)
  /sub/chat/room/{roomId}  (구독)
  메시지 수신 시 ChatMessage DB 저장 + 브로드캐스트

GET /api/chatrooms/me
GET /api/chatrooms/{roomId}/messages
  - 채팅방 목록/이력 REST 조회 (기존 REST 패턴 재사용)

[기존 코드에서 손봐야 할 부분]

- LoginCheckInterceptor의 startsWith 체크 문제 (위 "아쉬운 점 6"번과 동일)
  -> 신규 API는 경로에 "items"가 섞이지 않게 짓거나(/api/chat-rooms),
     이 기회에 AntPathMatcher 기반 화이트리스트로 교체 권장
- GlobalExceptionHandler에 "로그인은 했지만 이 채팅방/주문에 대한 권한
  없음" 케이스를 위한 403 개념 추가 필요 (현재 401만 존재)


================================================================
검증 방법 (각 단계 구현 후 확인용)
================================================================

앱 구동: .\gradlew.bat bootRun (+ H2 TCP 서버 별도 기동 필요)
ddl-auto=create 라서 재시작마다 스키마가 초기화되므로 H2 콘솔로
테이블 구조를 바로바로 확인 가능

- 단계 1 (Orders 상태전이): Postman/curl로
  로그인 -> 상품 구매 요청 -> PATCH로 상태 전이 API 호출
  -> Item.status와 Orders.orderStatus가 함께 바뀌는지 확인
  -> 잘못된 순서로 전이 시도 시 409가 오는지 확인
  -> H2 콘솔에서 ORDERS 테이블에 agreedPrice 컬럼이 생성됐는지 확인

- 단계 2 (위시리스트): 로그인 -> POST /api/wishlist/{itemId} 두 번
  연속 호출 시 추가/삭제가 토글되는지 확인 -> GET /api/wishlist로
  목록에 반영되는지 확인 -> 기존 프론트 Wishlist.jsx가 mock 모드가
  아닌 실제 API 모드에서도 정상 동작하는지 회귀 확인

- 단계 3 (검색): GET /api/items?keyword=...&category=...&priceGoe=...
  조합별로 curl/Postman 응답 확인 (이미 통과된 항목, 회귀 확인 목적)

- 단계 4 (채팅): 브라우저 기반 STOMP 테스트 클라이언트(stomp.js) 또는
  Postman의 WebSocket 요청 기능으로
  -> CONNECT 시 세션 쿠키 없이 거부되는지 확인
  -> 두 개의 클라이언트로 방을 열어 실시간 메시지 송수신 및 DB 저장이
     되는지 확인
